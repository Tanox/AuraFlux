# 贡献者行为规范

## 版本信息
- **版本**: v2.3.5
- **更新日期**: 2026-04-22
- **作者**: Sut

## 1. 开发工作流
- **分支**: `main` (稳定版), `dev` (集成版)
- **提交**: 严格执行 **Semantic Commits** (feat:, fix:, docs:, perf:)

## 2. 代码规范
- **版本说明**: 每个文件头部必须包含 `Version: vX.Y.Z` 注释
- **路径**: 优先使用 `@/` 路径别名
- **资源**: 禁止将静态资源放在 `app/` 目录

## 3. 规范更新
- 每次代码修改必须同步更新对应的 `openspec/` 文档，确保代码与文档的一致性

## 4. 代码审查
- **PR 流程**: 提交 PR 前必须运行测试和 lint
- **审查标准**: 代码质量、测试覆盖、文档完整

## 5. 测试要求
- **覆盖率**: 新增代码覆盖率 ≥ 80%
- **测试类型**: 单元测试、集成测试

## 6. 行为准则
- **尊重**: 尊重所有贡献者
- **包容**: 欢迎不同背景的贡献者
- **专业**: 保持专业的沟通方式

## 7. 安全注意事项
- **API 密钥**: 禁止在代码中硬编码 API 密钥
- **数据隐私**: 保护用户隐私
- **安全扫描**: 定期进行安全扫描

## 8. 文档规范
- **格式**: Markdown 格式
- **版本**: 与代码版本保持一致
- **语言**: 中文和英文双语文档

## 9. 问题报告
- **模板**: 使用提供的 issue 模板
- **信息**: 提供详细的问题描述和复现步骤
- **标签**: 使用合适的标签分类 issue

## 10. 功能请求
- **模板**: 使用提供的 feature request 模板
- **描述**: 详细描述功能需求和使用场景
- **影响**: 分析功能对现有代码的影响

## 11. 提交规范
- **提交信息**: 清晰、简洁、符合 Semantic Commits
- **变更范围**: 每个提交只包含一个逻辑变更
- **测试**: 确保所有测试通过

## 12. 发布流程
- **版本**: 遵循 Semantic Versioning
- **发布说明**: 编写详细的发布说明
- **测试**: 发布前进行全面测试

## 13. 社区参与
- **讨论**: 积极参与社区讨论
- **帮助**: 帮助其他贡献者
- **分享**: 分享使用经验和最佳实践

## 14. 责任
- **代码质量**: 对提交的代码负责
- **文档**: 确保文档的准确性
- **安全**: 关注代码安全

## 15. 奖励机制
- **贡献者名单**: 记录所有贡献者
- **表彰**: 对重要贡献进行表彰
- **机会**: 提供参与核心开发的机会
